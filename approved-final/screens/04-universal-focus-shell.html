Navigation verdict

One app. One focused action at a time. One-tap switch anytime.

The app opens in Social first. Mool temporarily changes the same dock into universal actions, Chat stays pinned for unread messages, and each selected action immediately opens its default useful flow.

Try focus switching

Main action words stay visible in the bottom dock. Tap Buy, Eat, Ride, Book, Pay or Work to open that world directly; the focused sub-actions appear inside the screen.

First impression

The default is engagement-first, not a static utility dashboard. The universal launcher uses direct action words while each focused action shows only its own relevant tabs.

Screen Contract Entry, behavior and exit

Entry Condition

Screen 03 has created a valid individual session and merged Screen 02 language, area and anonymous setup state.

Primary Behavior

Open one focused action at a time. Default action is Social with immediate Shorts consumption. The bottom rolling strip keeps main actions visible; focused sub-actions appear inside the active world.

Exit Paths

Each action opens its full feature flow: social activity, product buying, food, rides, bookings, payments, work or chat.

Role Layer Rule Consumer access remains

No Role Switch Tabs

Do not show Personal, Retailer, Taxi, Creator, Doctor or Manufacturer tabs on this screen. Main action words open product/service worlds, not manual user roles.

Upgrade Path

Profile can invite the user to add verified workspaces such as shop, clinic, creator, captain, service provider or manufacturer.

Automatic Panel Later

After a workspace is verified, relevant role actions can appear inside Social, Buy, Eat, Ride, Book, Pay, Work or Chat based on priority, time, location and pending work.

Full-Stack Implementation Contract What production must build

Frontend State

  • activeWorld: social, buy, eat, ride, book, pay, work or chat.
  • Bottom dock keeps main action words visible. Focused sub-actions appear inside the active world.
  • Universal action strip switches Social, Buy, Eat, Ride, Book, Pay and Work in one tap. Chat stays pinned separately.
  • Right-side action rail is reserved for current content actions such as like, comment, share, remix, save or apply.
  • Chat is one-tap inside the universal app strip and opens the chat stack.
  • Command bar routes direct intents into the correct world.
  • World state is restored on app resume unless a priority route exists.

Backend Bootstrap

  • Return user session, area state, language, enabled worlds and live service registry.
  • Return social feed bootstrap for default engagement.
  • Return serviceability flags for Buy, Eat, Ride and Book cards.
  • Return funded opportunity flags for Work.
  • Return payment, bill/recharge and receipt status for Pay.

Regression Gates

  • No visible role switch appears.
  • No unsupported service button appears.
  • Switching worlds removes distracting content from the previous world.
  • Consumer access remains even after workspace activation.
  • Screen 04 shows only MoolSocial as public app identity.
API Flow Screen 04 network contract
1. BootstrapGET /v1/app/focus/bootstrap
Input: userSession, device, language, area state, activeWorld preference and pending route.
Output: enabled modes, defaultMode, user profile summary, chat unread count and feature flags.
2. Social FeedGET /v1/social/feed
Input: userId, area, language, cursor, interest graph and safety filters.
Output: feed cards, reels, posts, creators, local updates and commerce attribution metadata.
3. Action CardsGET /v1/actions/decision-cards
Input: area, serviceability, user intent, active providers, live offers and compliance flags.
Output: decision-ready cards with price, time, trust/proof, payment/refund and next action.
4. Work BoardGET /v1/work/opportunities
Input: user verification, area, skills, workspace status and funded task registry.
Output: paid delivery work, onboarding work, verification work, workspace status, payout condition, review state and blocked reasons.
5. Pay SummaryGET /v1/pay/summary
Input: userId, service registry, settlement state and payment permissions.
Output: recharge, bill, scan pay, receipt status, refunds and payment support state.
What This Screen Must Not Do Regression-sensitive

No Static Dashboard

Do not fill the first screen with permanent utility buttons. The user should feel one active action at a time.

No Listing App

Do not show endless providers, products or categories. Show outcome-ready cards only when serviceable.

No Distraction During Focus

When user is in Social, commerce/work cards stay quiet. When user is in Buy, Eat, Ride, Book, Pay or Work, video/social content stays quiet.

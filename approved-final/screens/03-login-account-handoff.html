Screen Contract Entry, behavior and exit

Entry Condition

Screen 02 completed or skipped area setup. Anonymous device session, resolved language, area state and login options config are available.

Primary Behavior

Show one-method login. Social provider, email OTP or mobile OTP can each create or resume the same MoolSocial user identity.

Exit To Next Screen

After verified login, mint session, merge anonymous setup data into the user account and route to Screen 04. Screen 04 UI is intentionally not defined here.

Identity Rules Production decisions

Individual First

Login creates or resumes an individual user. Work, business, creator or captain layers are not requested here.

No Double Verification

Successful social login is sufficient unless risk rules demand recovery or step-up checks later. Email/mobile use OTP as their own path.

Account Linking

If the same email/mobile/provider maps to an existing account, resume it. If a new provider is added later, link only after verified consent.

Provider Screen Ownership Automatic vs MoolSocial-owned

External Screens

Google, Apple, X, Facebook and Instagram approval screens are native/provider-controlled surfaces. They appear automatically after tap and are not separate MoolSocial UI screens.

MoolSocial Screens

MoolSocial owns only method selection, OTP entry, retry/cancel return, blocked-provider messaging and post-callback route handling on Screen 03.

Auto Exit

After successful OAuth or OTP verification, Screen 03 should not show a separate success/loading page. It immediately routes to Screen 04 bootstrap.

Client State Machine Screen 03 states

Choose

Default state. Shows provider icons, Email OTP and Mobile OTP. No role selector, no personal/business toggle.

Provider Pending

Native/provider approval surface is active. On return, validate state nonce and provider result with backend; on cancel, return to method selection.

OTP Pending

Email or mobile OTP sent. User can verify, resend after cooldown, change method, or recover from expiry.

OTP Autofill

Mobile OTP fields support Android SMS Retriever/User Consent and iOS one-time-code autofill. Do not silently read the SMS inbox.

Session Ready

Backend returns user session, feature flags, area state and Screen 04 bootstrap. Frontend routes immediately without a visible loading screen.

API Flow Full-stack coding contract
1. Login OptionsGET /v1/auth/options
Input: anonymousSessionId, platform, locale, country, appVersion and area state.
Output: enabled providers, provider display order, provider route type native/oauth_browser/hidden, OTP cooldown rules, copy keys and route policy.
2. OAuth StartPOST /v1/auth/oauth/start
Input: provider, anonymousSessionId, redirectUri, PKCE challenge and anti-CSRF state.
Output: native provider payload or authorization URL, callback route, state nonce, provider timeout and cancel return route.
3. OAuth CallbackPOST /v1/auth/oauth/callback
Input: provider, authorization code or native token, PKCE verifier, state, nonce and device attestation.
Output: authResult success/cancel/denied/error, accountMatch, session token and nextRoute when successful.
4. OTP RequestPOST /v1/auth/otp/request
Input: channel email/mobile, normalized target, anonymousSessionId and risk context.
Output: otpAttemptId, masked target, expiry, resendAfter, attempt limit and allowed autofill mode.
5. OTP VerifyPOST /v1/auth/otp/verify
Input: otpAttemptId, code, autofillSource manual/os_sms/os_keyboard, anonymousSessionId, device fingerprint and locale.
Output: authResult, accountMatch, session token and nextRoute when successful.
6. Setup MergePOST /v1/auth/session/complete
Input: session token, anonymous setup state, area state, install source and pending deep-link.
Output: userSession, screen04Bootstrap, feature flags and recovery warnings if any.
Backend Data Model Tables or collections needed

User Identity

  • userId, status, createdAt, lastLoginAt and primary locale.
  • consumer profile is always active after login.
  • Role layers remain separate verified records later.

Identity Methods

  • provider, providerSubject, email hash, mobile hash and verification time.
  • One user can link multiple verified methods later.
  • Provider tokens are never trusted without backend validation.

Anonymous Merge

  • anonymousDeviceId, language, area, install source and pending route.
  • Merge must be idempotent to survive retry and app resume.
  • No duplicate user creation on repeated callback/OTP submit.
Security And Abuse Controls Non-negotiable

Rate Limits

Limit OTP requests by device, IP, target, area and attempt. Cooldowns must be visible and enforced by backend.

OAuth Safety

Use PKCE, state nonce, provider token verification, redirect allowlist and device/session binding.

Session Safety

Short-lived access token, refresh token rotation, device record and server-side session revocation.

Privacy

Store masked contact in UI. Raw contact data must be protected, hashed for lookup where possible and never logged in plain text.

Regression Checklist Before approval

Happy Paths

  • Google login succeeds and returns a Screen 04 route.
  • Email OTP succeeds and returns a Screen 04 route.
  • Mobile OTP succeeds and returns a Screen 04 route.
  • Phone OTP autofills into fields through supported OS autofill APIs where available.
  • Screen 02 area state is preserved after login.
  • Existing account resumes instead of creating duplicate user.

Edge Paths

  • Provider failure returns to method selection.
  • Expired OTP shows resend without losing selected method.
  • Too many attempts locks request briefly and suggests alternate method.
  • No role switch or business onboarding appears on this screen.
  • No verified role layer is created from email/mobile/social login alone.

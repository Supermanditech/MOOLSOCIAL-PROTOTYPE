Screen 66 separates failed payments from reversals before retry.

A failed attempt unlocks retry only after the bank confirms no debit. If money was debited, the same screen becomes a reversal tracker and blocks retry until the original amount is safely returned or reconciled.

Screen Contract Approved
EntryFrom Screen 63 failed outcome, Screen 65 final callback, payment alert, linked request/order or receipt history.
User SeesFailed/no-debit or debit/reversal state, amount, payee, purpose, method, attempt reference, bank timeline and valid next action.
DecisionRetry with another method only when no debit is confirmed. Reversal state permits status check and support, never retry.
BackendProvider error mapping, bank debit verification, idempotency key, reversal identifier, reconciliation queue, retry token and notifications.
Trust“Failed” never implies no debit until verified. Every reversal names its original destination and expected return time.
FailureUnknown debit returns to Screen 65 pending. Late success closes retry token; reversal delay opens support with references attached.
NextSafe retry returns to Screen 62 with the payee and purpose preserved. Reversal completion returns to Screen 64 receipt history.
RegressionNo callback, refresh or rapid tap can create duplicate attempts, expose retry during reversal, or overwrite an authoritative bank outcome.

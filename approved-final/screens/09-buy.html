UX Verdict

Screen 09 should default to grocery because it is a high-frequency household use case. But Buy must remain bigger than grocery: categories, medicine and repeat basket stay one tap away without changing the MoolSocial universal pattern.

Decision Card Rule

The backend may evaluate thousands of shops, suppliers, manufacturers, stockists and source partners, but the consumer sees only serviceable cards that answer price, fulfilment time, trust proof, payment option, refund/cancellation and next action.

Campaign Wording

Do not show "campaign engine" to consumers. Surface it as Buy Retail, Family Pack, Wholesale MOQ, Shop Offer or Ready Offer. The campaign/demand-pool logic remains backend architecture.

Medicine Boundary

Medicine appears only in active compliant areas with licensed pharmacy workflow, pharmacist approval, invoice, batch/expiry record and delivery proof. No open medicine marketplace.

Screen Contract What Screen 09 must preserve
EntryFrom Screen 04 Buy action or Mool dock root action.
Default content opens on Grocery with local decision cards, not a product listing page.
If entry is a shop QR, the app opens a saved shop session after install/login and keeps purchase actions tied to that verified shop.
QR Walk-InStore-specific purchase.
QR payload carries shop id, branch id, campaign/ref token and optional basket intent. If app is not installed, Play Store/referrer stores the pending shop session.
After app open, location permission or repeat counter QR confirms whether user is near that shop. If matched, show Buy Here, Scan Item, Counter Pickup and Home Delivery before generic local options.
Location MatchConsent-first logic.
Do not silently collect precise location during download. Use install referrer for shop identity, then verify distance after consent, approximate area, Wi-Fi/shop beacon if available or another QR scan.
If user is not near the saved shop later, keep the shop session as saved and show nearby serviceable shops separately. Never force the old shop context when the user is at another place.
First BehaviorShow matched outcomes.
The first card should show exact product lines, quantity, quality/brand, line price, total, ETA, proof, payment and refund rule.
Search, scan and voice are always available because many users will speak a list, scan an offline bill or compare brand/quality before ordering.
Provider LogicHide supply complexity.
The UI never asks the consumer to compare 10,000 shops or manufacturers manually.
Backend ranks supply by location, stock, wholesale target, proof, fulfilment capacity, backup availability, complaint risk and margin.
Demand LogicUser wording stays simple.
User sees clear purchase choices: Buy Retail, Family Pack, Wholesale MOQ and Shop Offer.
Wholesale Basket must show target progress and fallback maximum price before user joins. Backend may create a demand pool, supplier opportunity, source campaign, delivery task or wholesale workflow.
ComplianceMedicine is gated.
Medicine does not appear where a licensed pharmacy/vendor workflow is not active.
Prescription, pharmacist approval, invoice, batch/expiry, return policy and delivery proof are API-level gates.
RegressionDock remains usable.

Full-Stack Implementation Notes For later production build

Decision API

  • Input: user area, intent, cart/list, category, urgency and payment readiness.
  • Output: small set of ranked decision cards, not provider directory results.
  • Every card must expose price, ETA, proof, payment, refund/cancel and next action.

Supply Graph

  • Rank shops, distributors, source partners and fulfilment routes by readiness.
  • Support backup supplier rules before promising delivery.
  • Do not expose manufacturers directly unless category and fulfilment permit it.

Demand Pool

  • Wholesale Basket or Saved Basket can create demand signals behind the scenes.
  • Campaign, demand wallet and ledger are backend terms.
  • User-facing states must stay simple: Buy Retail, Family Pack, Wholesale MOQ, Shop Offer, Save Repeat.
  • Wholesale MOQ needs minimum quantity, current joined quantity, wholesale rate and fallback retail price.

Safety Gates

  • Medicine requires licensed pharmacy activation and pharmacist workflow.
  • High-value products require invoice, return, warranty and dispute rules.
  • Payment capture should occur only when fulfilment dependency is ready enough.

Screen 02: First Setup / Language + Location

This screen removes unnecessary setup choices. MoolSocial uses the phone language automatically, explains why area matters, asks for phone location only after one clear tap, and treats area as a current/default choice instead of assuming the first detected city is home. If the user is travelling, they can set their home/default area through search, address, pin code or map. If the phone language is not supported yet, the app continues with a safe fallback and exposes language change later.

What happens after choice

Screen 02 has three actions. Current location asks phone permission, manual area opens search or map, and skip continues to login.

  • 1Use current location: phone asks permission, then MoolSocial saves current area and continues to account sign in.
  • 2Set manually: user searches area, pin code or address, or chooses on map. This can become the default/home area.
  • 3Skip now: continue to account sign in with area unset. MoolSocial asks again after signup before location-based results or actions.
  • 4Next: Screen 03 opens the sign-in method screen. Login provider choices and OTP belong only to Screen 03.

Current location path

Saved as current area only. It is not treated as permanent home unless the user confirms later.

Manual path

User can type, search, use pin code, or pick on map. This supports users who install while travelling.

Skip now path

Current area stays unset, login choice still opens, and MoolSocial asks again after signup when location-based results or actions need it.

Screen Contract Entry, behavior and exit

Entry Condition

Fresh install exits Screen 01 boot successfully and reaches first setup with device locale, timezone, install source and permission state available.

Primary Behavior

Auto-select language, show area setup choices, support current-location/manual/skip, and avoid all role selection.

Exit To Next Screen

After current area, manual area or Skip now path, route to Screen 03: Login / Account Handoff. If phone permission is not allowed after tapping Use current location, treat it like Skip now.

Language Logic No first-open picker

Auto Select

Use the phone's preferred language, region and script. If the exact language is supported, it becomes the app language immediately.

Unsupported Phone Language

If the phone language is not supported, use English India as the safe default. Do not block setup and do not force a language decision.

Content Layer

System UI, category labels, trust messages and local help copy follow the selected app language. Seller/product names are not force-translated.

Language Coverage Architecture India-wide without launch delay

Target Coverage

Build the platform so all major Indian territory languages can be added as language packs, including the 22 scheduled Indian languages over time.

MVP Launch Rule

Launch only with professionally checked UI languages. Missing languages must fall back cleanly instead of showing broken or low-trust translation.

Later User Control

After first setup, keep language change in profile/settings and possibly in help. This follows large social-app behavior without slowing onboarding.

Travel And Home Handling Important edge case

User Downloads While Travelling

  • If the user chooses current location while travelling, save it only as current area, not permanent home.
  • If the user wants home/default area while travelling, they use Set manually through search, pin code, address or map picker.
  • Screen 04 must later decide the exact active-area and correction UI.
  • Delivery address, pickup point, ride pickup and work site are confirmed at the transaction moment.

Refresh Policy

  • Refresh current area on app open, app resume and location-based action start when permission is available.
  • Do not overwrite manually selected home/default area without user confirmation.
  • Use low-frequency foreground checks, not constant tracking.
  • Background location is allowed only for active ride, delivery, field work or safety-critical task with separate consent.
  • If location is stale, show a one-tap refresh before showing decision-ready local cards.

Full-Stack Implementation Contract What production must build

Frontend UI Modules

  • FirstSetupScreen owns the full screen route after splash.
  • LanguageAutoResolver silently selects phone language or fallback without adding a first-open choice.
  • LocationValueCard explains area use before consent without tying the app to one user type.
  • LocationConsentButton opens the native phone permission prompt.
  • ManualAreaButton opens search, pin code, address and map picker.
  • ServiceAreaPreview shows current area, manual area, stale area, skipped state or low-confidence correction.
  • ContinueRouteGuard decides when to move to Screen 03 Login / Account Handoff.

Client State Machine

  • bootReady: received Screen 01 boot session and device metadata.
  • languageResolved: app language chosen from device or fallback.
  • locationExplained: user sees why location matters.
  • areaMethodSelected: gps, manual or skip.
  • permissionRequested: only after one tap on Use current location.
  • manualAreaSearch: user searches city, locality, pin code or address.
  • serviceAreaResolved: current or manual area saved with confidence and expiry.
  • routeReady: Login / Account Handoff can load with location outcome and local readiness flags.

Backend Ownership

  • Onboarding service owns setup session and route decision.
  • Localization service owns supported language packs and fallback order.
  • Location service owns current-location reverse geocode, manual search/map resolve, confidence and service area.
  • Serviceability service owns which local features are available in the area.
  • Consent service records permission state without forcing access.
API Flow Screen 02 network contract
1. Bootstrap SetupPOST /v1/onboarding/first-setup/bootstrap
Input: bootSessionId, anonymousDeviceId, appVersion, preferredLocales, timezone, install source, current permission state.
Output: resolved language, fallback reason if any, location copy keys, route policy and feature readiness flags.
2. Resolve LanguagePOST /v1/onboarding/language/resolve
Input: preferredLocales, region, script, supportedLanguagePackVersion and app channel.
Output: languageCode, languageSource, fallbackUsed, copyBundleUrl and unsupportedLocaleRecorded.
3. Resolve Current AreaPOST /v1/location/current-service-area
Input: consentGrantId, approximate/precise flag, latitude/longitude after consent, device timestamp and accuracy.
Output: country, state, city, locality, pin code, confidence, precision, expiry and serviceability by feature.
4. Resolve Manual AreaPOST /v1/location/manual-area/resolve
Input: searchText, pinCode, addressText or mapPin, plus selected area type: default/home candidate.
Output: normalized area, map bounds, confidence, serviceability by feature and display label.
5. Skip Or Permission Not AvailablePOST /v1/onboarding/location/outcome
Input: skipNow, permissionNotAllowed, approximateOnly or promptUnavailable, plus screen source and timestamp.
Output: login route, area-unset state and which later local actions must ask again in context.
6. Complete SetupPOST /v1/onboarding/first-setup/complete
Input: languageCode, serviceAreaId if available, location permission state and current route intent.
Output: nextRoute, loginOptionsConfig, local feature flags and pending deep-link resume target.

Privacy, Security And Observability Non-negotiable production rules

Privacy

Precise phone location is collected only after consent. Manual search/map can save a user-chosen home/default area. General browsing stores area, not constant movement.

Security

All setup APIs require device attestation or signed anonymous session from Screen 01. Rate-limit location updates and reject impossible jumps or stale timestamps.

Monitoring

Track setup completion rate, permission grant rate, fallback language rate, low-confidence geocode rate, API latency and route-to-Screen-03 failures.

What This Screen Must Not Do Regression-sensitive

No Language List

Do not force Hindi, English, Gujarati, Marathi or any language choice during first setup. Phone language drives setup; fallback is automatic.

No Auto Home Lock

Do not store the first detected city as permanent home. Only user manual selection can create a home/default area.

No Always-On Tracking

Do not track live location in the background for general browsing. Use foreground refresh and task-based tracking only.

Regression Checklist Before approval

Happy Path

  • Phone language auto-selects without showing a picker.
  • Unsupported phone language falls back to English India without blocking setup.
  • Area setup is explained before any OS permission prompt.
  • Use current location creates a current-area record.
  • Set manually supports search, pin code, address and map picker.
  • High-confidence area routes to Login / Account Handoff.
  • No role switch appears anywhere.

Edge Paths

  • Unsupported phone language falls back without blocking setup.
  • Skip now keeps app usable and routes to login.
  • Travel install with current location does not become permanent home location.
  • Manual home/default area is not overwritten by later current-area refresh without confirmation.
  • Precise location is asked only for delivery, ride, pickup, work or fulfilment moments.
